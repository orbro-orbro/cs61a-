# -Ants Vs. SomeBees

这是一个用 Python 实现的塔防小游戏。整体玩法是：蚂蚁在 colony 中布防，蜜蜂从蜂巢出发沿着 tunnel 推进，双方通过移动、攻击、阻挡和特殊技能进行对抗。

## 这个项目实现了什么

这份代码的核心是在一个完整的游戏框架里，组织出一套可扩展的昆虫战斗系统。它实现了以下几类能力：

- 地图与路径管理：用 `Place` 把一条条 tunnel 串起来，并维护 `exit` / `entrance` 关系。
- 昆虫基础机制：所有蚂蚁和蜜蜂都继承自 `Insect`，共享血量、位置、死亡处理等通用行为。
- 蚂蚁分工系统：不同蚂蚁承担不同职责，例如产粮、远程攻击、近战反伤、容器保护、坦克防御、女王增益等。
- 蜜蜂行动系统：蜜蜂可以移动、攻击、进入水域限制、受到控制效果影响。
- 游戏流程控制：`GameState` 负责回合推进、资源管理、胜负判断和布局配置。

## 你在这里实现的功能风格

这份代码体现的是一套典型的面向对象设计：

- 使用类继承复用公共逻辑。
- 使用类属性描述角色类型差异，比如 `food_cost`、`damage`、`implemented`、`is_container`。
- 使用实例属性保存局部状态，比如血量、当前位置、冷却时间、是否已经强化过。
- 使用方法重写定制每种角色的行为，比如 `action`、`reduce_health`、`add_to`、`blocked`。
- 使用组合关系表达复杂对象，比如容器蚂蚁可以“装下”另一只蚂蚁。

## 文件整体框架

这份文件可以大致分成几层：

### 1. 核心世界模型

这一层定义了游戏里的基础对象：

- `Place`：地图上的格子，负责连接前后位置并保存蚂蚁和蜜蜂。
- `Insect`：所有昆虫的共同父类，提供生命值、位置、受伤和死亡逻辑。
- `Ant`：蚂蚁的基类，扩展出放置规则、容器逻辑和增益逻辑。

### 2. 普通蚂蚁与特殊蚂蚁

这一层定义各种具体单位：

- `HarvesterAnt`：负责产出食物。
- `ThrowerAnt`：负责向前寻找最近的蜜蜂并攻击。
- `ShortThrower` / `LongThrower`：限制攻击距离的远程单位。
- `FireAnt`：受到伤害时会对同格蜜蜂造成反伤。
- `WallAnt`：只负责承伤，不主动行动。
- `HungryAnt`：吞掉一只蜜蜂后进入咀嚼冷却。
- `ContainerAnt` / `ProtectorAnt` / `TankAnt`：支持“一个格子里放两只蚂蚁”的容器体系。
- `ScubaThrower`：可以在水里生存的远程蚂蚁。
- `QueenAnt`：提供全局式的攻击增幅，并负责特殊胜负判定。

### 3. 蜜蜂与蜂巢

这一层负责敌方单位的行为：

- `Bee`：会向前移动或攻击阻挡它的蚂蚁。
- `Wasp` / `Boss`：更强的蜜蜂变体。
- `Hive`：蜜蜂的出生点，也是敌方生成入口。

### 4. 游戏运行时

这一层管理整局游戏的执行：

- `GameState`：保存当前回合数、食物、所有地点、活动蜜蜂等状态。
- `simulate`：控制每一回合的完整流程。
- `deploy_ant`：根据食物和蚂蚁类型，把单位部署到地图上。
- `ants_take_actions` / `bees_take_actions`：分别处理双方行动。

### 5. 布局与生成

这一层把地图组织成可玩关卡：

- `wet_layout` / `dry_layout`：生成不同地形。
- `AssaultPlan`：描述蜜蜂在什么时间、以什么数量出现。

## 设计重点

这个项目最值得关注的地方，不是单个函数，而是这些功能如何互相配合：

- 地图连接决定了蜜蜂如何前进，蚂蚁如何追踪敌人。
- 放置规则决定了一个格子能容纳几只蚂蚁、什么类型的蚂蚁可以共存。
- 伤害和死亡逻辑决定了单位如何退出战场。
- 增益、反伤、冷却、水域、状态效果这些机制，都是在同一套基类之上逐步叠加出来的。

## 一句话总结

这是一个围绕“继承 + 组合 + 回合制状态管理”构建出来的塔防项目，核心价值在于把一套基础战斗框架扩展成多个角色、多种规则和多层交互的完整小游戏。
